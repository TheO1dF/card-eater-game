| 常规类别 | 9 类：水果、快餐、甜点、饮料、蔬菜、星体、人物、动物、通用 |
| 特殊类别 | 无类别：虚空牌 |
| 底层效果类型 | 60 种，最大单一类型组 7 张 |
| 永久规则 | 74 条，同一局不重复 |

| 任务道具 | 12 件 |

| 阶段累计目标 | 第 5 轮 150、第 10 轮 1500、第 15 轮 12000 |

核心设计仍是“小牌组可稳定完成位置组合和重洗成长，大牌组有较高基本分但会稀释组合”。经济与得分故意分流：吃牌提供基础金币；纯弃牌需要专门经济卡或道具；摧毁既可以压缩牌组，也可以转为金币、折扣、生成或永久成长。

## 状态机与一轮流程

严格状态迁移定义在 `js/state.js`：

```text
Init
  → RuleDraft
  → QuestDraft（只在任务轮出现）
  → Playing
  → Scoring
  → Shop
  → NextRound
  → RuleDraft / GameOver
```

`js/main.js` 只负责编排阶段、输入锁、动画等待和 UI；`js/engine.js` 不访问 DOM，是所有吃/弃、卡牌效果、规则判定和结算的核心。不要把具体卡牌 ID 的特判写进 UI 或主流程，应继续通过 `effect.kind` 扩充解析器。

永久状态放在 `state.deck`、`active_rules`、`items`、任务历史和待生效奖励中；轮内状态放在 `state.round`。重要轮内计数包括：

- `actions`、`eat_sequence`、`discard_sequence`
- `buffs`、`final_multipliers`
- `destroyed_count`、`generated_count`、`grown_count`
- `reshuffle_charges`、`reshuffle_count`
- `pending_gold_bonus`、`shop_discount`
- `shop_reroll_count`、`shop_free_rerolls`
- `effect_trigger_counts`：限制“每轮首次”等效果

卡牌结算顺序是：即时净化 → 永久规则平加 → 消耗旧蓄势 → 读取牌面 → 道具/任务修正 → 执行本牌效果 → 本次总分 → 写入行动历史。新蓄势只能影响之后的牌，不能回头强化本牌。

## 统一关键字契约

`js/keywords.js` 是唯一术语入口。数据载入时会为效果附加关键字并把描述规范化；UI、商店和 CSV 使用规范化后的描述。

当前玩家关键字：

- 【摧毁】：从本局永久牌组移除；重洗也不会回来。
- 【相邻】：读取初始牌序中本牌前后紧挨的牌。
- 【成长】：永久提高指定实体牌的吃分或弃分。
- 【生成】：向永久牌组加入带 `generated_from` 的新实体牌。
- 【重洗】：回收本轮已处理且仍存在于永久牌组的牌。
- 【净化】：清除尚未结算的负面蓄势，不返还已扣分数。
- 【追溯】：读取本轮已经发生的行动或得分。
- 【蓄势】：保存给后续符合条件牌的效果。
- 【位置】：检查首位、次位、中位、末位或行动序号。
- 【规模】：读取永久牌组数量、类别占比或不同卡名。
- 【复制】：复制数值，不复制原卡效果。
- 【经济】：改变金币、商店价格、刷新或回收收益。
- 【储存】：把数值保存在具体实体牌上，跨轮保留。
- 【预判】：读取尚未处理的牌，不改变牌序。

所有从永久牌组移除卡牌的描述必须只使用“摧毁”。不要重新引入其他近义词。成长描述必须包含准确的 `+N`，或说明 `+X` 的来源、上下界、成长属性。例如狼按稀有度获得弃分 `+1/+2/+3/+5`，噬牌虎按猎物吃分绝对值成长 `1～4`。

## 卡牌差异化现状

重做后不再只有“换目标类别和数字”的一套模板：109 张有效果卡覆盖 60 种底层效果类型。仍然共用同一种基础解析器的牌，至少在行动侧、历史窗口、生成条件、目标范围、位置成本或资源产出上存在差异。

代表性机制：

- 草莓读取连续水果，水果拼盘读取不同水果名，贪吃狗读取不同可食用卡名，猴子读取紧邻水果连段，饕餮只追溯主动吃下的负分牌。
- 汽水让下一张使用相反牌面；巨无霸持续惩罚后续吃牌直到任意弃牌；国王让下一张正牌面 ×5、非正牌面额外 -5。
- 糖果把分数储存在这张实体牌上，吃时累积、弃时兑现并清零。
- 南瓜检查蔬菜是否占牌组半数；星图检查不同星体卡名；兔群头领按兔子对子而非单纯数量结算。
- 果核无条件生成种子；外卖袋要求末位并一次生成两张热狗；空瓶生成牛奶后摧毁自身；母鸡要求紧接动物；园丁要求紧接弃掉的蔬菜；复印纸要求预判后一张不可食用牌。
- 母鸡生成的鸡蛋有额外 +4，商店购买的鸡蛋没有，生成来源是构筑资源。
- 松鼠摧毁上一张可食用牌换金币；清道夫摧毁上一张牌换商店折扣；蔬菜汤摧毁上一张蔬菜换永久成长。
- 堆肥箱把此前弃食节奏转为金币，不再只是又一张历史加分牌。

完整卡表以 `js/data.js` 和自动导出的 `cardeater.csv` 为准。CSV 已包含持有上限、关键字、效果类型和完整描述。

## 第 10 轮无法购买问题

旧设计存在两类风险：牌组安全上限偏小，以及商店可能展示已经达到持有上限、实际无法购买的商品。免费刷新后如果 UI 仍持有旧商品或只显示统一错误，也容易被误认为“金币不足”。

当前防线在 `js/shop.js`、`js/main.js` 和 `js/config.js`：

1. 牌组上限由 80 提高到 160，单轮行动上限同步提高到 400。
2. 所有卡牌都有持有上限：传奇默认 1、普通默认 3、其他默认 2；卡牌可显式覆盖，兔子为 12。
3. `getShopCards()` 在抽商品前过滤达到上限的 ID，避免死货位。
4. 购买前仍由 `getBuyCardStatus()` 二次检查金币、牌组上限、卡牌存在性与持有上限。
5. UI 分别显示 `insufficient_gold`、`deck_full`、`copy_limit`、`invalid_offer`，不再一律报金币不足。
6. 刷新会同时替换卡牌和道具缓冲；免费刷新照样增加后续刷新价格。
7. 单测覆盖“免费刷新后购买”和“第 10 轮、100 张牌组、金币 100 时购买成功”。

如果线上仍报告同类问题，优先确认部署是否确实包含 v0.8，并记录商品 ID、牌组张数、同 ID 数量、金币和控制台堆栈；不要先放宽金币判断。

## 随时查看牌组与任务

`index.html` 增加浮动“组”按钮和 `#deckStatus`；`js/ui.js` 的 `openDeckStatus()` 会按“卡牌 ID + 当前吃/弃分 + 生成来源 + 储存分 + 成长进度”分组，因此成长程度或生成来源不同的同名实体牌不会被错误合并。

牌组查看器展示：

- 当前张数 / 160 与类别分布
- 卡图、数量、ID、稀有度、类别、食用属性
- 当前吃分、弃分
- 储存分与成长进度
- 生成来源（例如母鸡生成的鸡蛋）
- 规范化后的完整效果
- 可展开的关键字词典

“任”按钮继续显示当前任务的惩罚、完成要求、奖励和“下一轮生效”提示；结算栏明确显示完成/失败及生效轮次。牌组与任务浮层高于规则、结算和商店，但低于欢迎页和倒计时。

```powershell

```

有浏览器调试环境时，再运行烟测并确认：

- 欢迎页、规则选择、任务选择、游玩、结算、商店均无横向溢出
- “组”按钮能打开牌组，卡牌效果数与展示卡数一致，关键字词典不为空
- “任”按钮能显示要求、惩罚、奖励和下一轮生效说明
- 免费刷新后仍可购买，金币按商品价格扣除
- 手机滑动和桌面按钮都能完成前三轮
- `browser_errors` 为空

平衡继续测试时，应记录路线、牌组大小、金币来源、摧毁/生成/成长次数、吃弃比例、每轮耗时和规则完成率。最需要人工试玩的风险是：

1. 60 种效果提高了理解成本，关键字虽统一，但单张描述仍可能偏长。
2. 牌组上限 160 是安全边界，不代表 100+ 张牌的操作时长已经适合真人。
3. 摧毁换经济的松鼠/清道夫与免费刷新、折扣道具叠加后可能过强。
4. 生成来源加分需要确认 UI 是否足够让玩家理解“商店鸡蛋”和“母鸡鸡蛋”的区别。
5. 道具图标复用了 4×4 元图集坐标，功能完整，但若追求视觉唯一性需要另做美术扩容。
